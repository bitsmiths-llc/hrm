# Domain Blockers — BitSmiths HRM

A **blocker** is a defect that must be fixed before merge. These are the hard rules for
this repo, derived from `CLAUDE.md`, `.claude/skills/coding-standards/SKILL.md`, and the
`.claude/docs/rules/*` / `docs/backend/*` guides. Each maps to one line on the Domain
Blockers checklist in `docs/review/review-protocol.md`.

When in doubt, prefer flagging over silence — but only tier something `[BLOCKER]` if it
matches one of the rules below. Everything else is `[MAJOR]` / `[MINOR]` / `[NIT]`.

---

### 1. Authentication gate on every authenticated entry point

- Authenticated **server actions** must be built with `authActionClient`
  (`src/lib/server/safe-action.ts`), not `safeActionClient`. Public actions use
  `safeActionClient` deliberately.
- Authenticated **queries** must run through `authQuery()`
  (`src/lib/client/auth-query`), which enforces the session (and role) before the handler.
- A new action/query that reads or writes user-scoped data **without** an auth gate is a
  BLOCKER. See `docs/backend/auth-and-supabase.md`.

### 2. Service-role key stays server-only

- The Supabase service-role key bypasses RLS. It may only be used on the server and must
  **never** appear in the browser client, a `'use client'` module, or any code reachable
  from one. Exposing it is a BLOCKER.
- RLS stays enabled by default; do not disable it to "make a query work."

### 3. Server/client boundary is not crossed

- Server-only modules — `src/lib/supabase/server.ts`, everything under `src/lib/server/**`
  (including `safe-action.ts`), and any service-role client — must never be imported by a
  `'use client'` file or a module a client component imports. See
  `.claude/docs/rules/server-client-components.md`. A crossed boundary is a BLOCKER
  (it leaks server code/secrets into the client bundle).

### 4. Environment variables only via `src/env.ts`

- `process.env.X` must not be read directly anywhere except `src/env.ts` (t3-env). Direct
  access elsewhere bypasses validation and can leak server secrets to the client bundle —
  BLOCKER. See `.claude/docs/rules/config-and-routes.md`.

### 5. Generated Supabase types are never hand-edited

- `src/types/supabase.ts` is generated by `pnpm supa:types`. Hand-edits are a BLOCKER —
  they drift from the schema and get overwritten. Regenerate instead.

### 6. Migrations are paired with regenerated types

- Any change under `supabase/migrations/**` must be accompanied by a regenerated
  `src/types/supabase.ts` (`pnpm supa:types`). A migration that changes the schema without
  updating the generated types is a BLOCKER (runtime/type mismatch).

### 7. shadcn primitives are added, not hand-written

- Components under `src/components/ui/*` come from `pnpm ui add <component>`. Hand-editing
  or hand-authoring them diverges from the shadcn source and breaks upgrades — BLOCKER
  (per `CLAUDE.md`). Compose wrappers in your own component files instead.

### 8. No secrets committed

- No `.env*` values, Supabase keys, or other credentials in the diff. A committed secret
  is a BLOCKER — rotate the key and remove it from history.

---

## Not blockers (flag at the right tier instead)

- `any` / `as any` / `@ts-ignore` without a `// reason:` comment → `[MAJOR]`
  (`[BLOCKER]` only at a trust boundary where it defeats validation)
- Inline query-key arrays instead of the central `QueryKeys` factory → `[MINOR]`
- kebab-case / naming / file-placement violations (`.claude/docs/rules/file-placement.md`) → `[MINOR]`
- Leftover `Logger.*` / `console.*` debug logs → `[MINOR]`
- Missing `onError` toast wiring on a mutation → `[MAJOR]` (user-facing failure is silent)
- Style preferences not encoded in a rule → `[NIT]`
